# ISChaos-sub 第二代混沌工程演练平台 — 需求分析文档

> 版本：v1.0 | 更新日期：2026-03-12

---

## 1. 项目背景与目标

### 1.1 项目背景

在微服务架构中，服务间的调用关系复杂，单一服务故障可能引发级联失败，导致整体系统不可用。混沌工程（Chaos Engineering）通过主动注入故障，在受控环境中验证系统的容错能力和弹性恢复机制。

本项目以 **Train-Ticket** 微服务系统为目标，构建一套基于业务链路的第二代混沌工程演练平台，提供故障注入配置、执行、状态监控与观测数据收集的全流程自动化能力。

### 1.2 项目目标

| 编号 | 目标 | 说明 |
|------|------|------|
| G-01 | 链路可视化 | 展示 15 个核心业务接口的调用链拓扑，包括服务间 HTTP 调用和数据库依赖 |
| G-02 | 故障注入自动化 | 支持 10 种故障类型的参数化配置，自动生成并提交 Chaos Mesh Workflow |
| G-03 | 执行状态监控 | 实时跟踪演练执行状态，展示故障注入/恢复的时间线 |
| G-04 | 观测数据收集 | 在故障注入前后自动采集 Traces、Metrics、Logs 观测数据 |
| G-05 | 故障指标验证 | 根据故障类型自动关联对应的可观测指标（如延迟、丢包率、CPU 使用率），以图表形式展示故障效果 |

## 2. 用户角色定义

| 角色 | 描述 | 典型操作 |
|------|------|---------|
| **混沌工程师** | 主要用户，负责设计和执行混沌实验 | 查看链路拓扑 → 配置故障计划 → 执行演练 → 查看结果 → 收集数据 |
| **SRE / 运维工程师** | 观察故障对系统的影响，评估系统弹性 | 查看执行记录 → 分析指标图表 → 下载观测数据 |
| **系统管理员** | 负责平台部署和维护 | 部署平台到 K8s → 配置外部服务地址 → 监控平台运行状态 |

## 3. 功能需求列表

### 3.1 链路拓扑管理

| 编号 | 需求 | 优先级 | 说明 |
|------|------|--------|------|
| FR-101 | 查看接口列表 | P0 | 展示 15 个业务接口的概要信息：名称、HTTP 方法、URL、服务数量、边数量、请求总数 |
| FR-102 | 查看链路拓扑图 | P0 | 以有向图（Dagre 分层布局）可视化接口的完整调用链，区分服务节点、根节点和数据库节点 |
| FR-103 | 拓扑交互 | P1 | 支持缩放、平移、节点悬停高亮上下游路径、点击服务节点打开故障配置 |
| FR-104 | 拓扑数据初始化 | P1 | 从 topology_report.md 文件自动解析并导入拓扑数据，支持 MongoDB 节点拆分 |

### 3.2 演练计划管理

| 编号 | 需求 | 优先级 | 说明 |
|------|------|--------|------|
| FR-201 | 创建演练计划 | P0 | 从拓扑页面点击服务节点，配置故障类型、参数和持续时间，创建演练计划 |
| FR-202 | 查看计划列表 | P0 | 展示所有演练计划，包含名称、故障类型、目标服务、时长、状态、创建时间 |
| FR-203 | 预览 Workflow | P1 | 在执行前预览将要提交给 Chaos Mesh 的 Workflow JSON |
| FR-204 | 修改计划 | P1 | 支持编辑计划的名称、参数、状态等字段 |
| FR-205 | 删除计划 | P1 | 支持删除不再需要的演练计划 |
| FR-206 | 计划状态管理 | P0 | 计划有 draft（草稿）、ready（就绪）、archived（归档）三种状态，仅 ready 状态可执行 |

### 3.3 演练执行管理

| 编号 | 需求 | 优先级 | 说明 |
|------|------|--------|------|
| FR-301 | 执行演练 | P0 | 一键执行 ready 状态的计划，自动构建 Workflow 并提交到 Chaos Mesh |
| FR-302 | 查看执行列表 | P0 | 展示所有执行记录，包含关联计划名称、故障类型、目标服务、状态、时间 |
| FR-303 | 执行详情页 | P0 | 展示执行的完整信息：状态时间线、计划参数、Chaos Mesh 工作流状态 |
| FR-304 | 实时状态同步 | P0 | 自动轮询 Chaos Mesh Workflow 状态（运行中每 5 秒刷新），同步更新本地记录 |
| FR-305 | 停止执行 | P0 | 支持手动停止正在运行的演练，调用 Chaos Mesh 停止 API |
| FR-306 | 删除执行记录 | P1 | 支持删除已完成/失败的执行记录（运行中的不允许删除） |
| FR-307 | 故障指标图表 | P0 | 在执行详情页展示与故障类型对应的可观测指标时序图（ECharts），观测窗口为故障注入前后各 10 分钟 |

### 3.4 观测数据收集

| 编号 | 需求 | 优先级 | 说明 |
|------|------|--------|------|
| FR-401 | Traces 采集 | P0 | 收集目标服务在故障期间的分布式调用链数据 |
| FR-402 | Metrics 采集 | P0 | 收集目标服务在故障期间的监控指标数据（CPU、内存、网络、延迟等） |
| FR-403 | Logs 采集 | P0 | 收集目标服务在故障期间的结构化日志数据 |
| FR-404 | 数据下载 | P0 | 每个数据类别（Traces/Metrics/Logs）独立的 JSON 下载按钮，支持自动获取并下载 |
| FR-405 | 拓扑查询 | P1 | 通过 Kubernetes API 获取实时的 Pod/Service/Node 部署拓扑 |
| FR-406 | 实体查询 | P1 | 获取集群中的 Pod、Service、Node 实体列表，支持按类型过滤 |
| FR-407 | 故障状态查询 | P1 | 查询当前活跃的故障锁定状态 |

### 3.5 故障类型支持

| 编号 | 故障类型 | Chaos Mesh 类型 | 说明 |
|------|---------|----------------|------|
| FR-501 | network_delay | NetworkChaos | 网络延迟注入，可配置延迟时长、抖动、相关性 |
| FR-502 | network_loss | NetworkChaos | 网络丢包注入，可配置丢包率、相关性 |
| FR-503 | network_partition | NetworkChaos | 网络隔离，可配置方向（双向/单向） |
| FR-504 | pod_failure | PodChaos | Pod 故障模拟，使 Pod 不可用 |
| FR-505 | pod_kill | PodChaos | 终止 Pod 进程，可配置优雅终止时间 |
| FR-506 | stress_cpu | StressChaos | CPU 压力注入，可配置并发 Workers 和负载百分比 |
| FR-507 | stress_mem | StressChaos | 内存压力注入，可配置并发 Workers 和内存大小 |
| FR-508 | dns_error | DNSChaos | DNS 解析错误注入，可配置匹配模式 |
| FR-509 | node_cpu | PhysicalMachineChaos | 物理节点 CPU 压力，需指定节点地址 |
| FR-510 | node_mem | PhysicalMachineChaos | 物理节点内存压力，需指定节点地址 |

## 4. 非功能需求

### 4.1 故障互斥约束

| 编号 | 需求 | 说明 |
|------|------|------|
| NF-01 | 服务级互斥锁 | **同一服务在同一时间只允许存在一个活跃故障**。执行前需获取服务锁，获取失败返回 HTTP 409 |
| NF-02 | 锁自动释放 | 演练完成（正常/超时/手动停止）时自动释放服务锁 |
| NF-03 | 残留锁清理 | 删除执行记录时同步清理可能残留的故障锁 |

### 4.2 可靠性

| 编号 | 需求 | 说明 |
|------|------|------|
| NF-04 | 超时保护 | 执行时长超过计划 duration 的 3 倍时自动判定为超时失败，释放锁并标记状态 |
| NF-05 | API 错误处理 | 所有外部 API 调用（Chaos Mesh、Coroot、K8s）均有超时控制和错误捕获 |
| NF-06 | 状态一致性 | 通过 sync_execution_status 机制确保本地执行状态与 Chaos Mesh 实际状态一致 |

### 4.3 性能

| 编号 | 需求 | 说明 |
|------|------|------|
| NF-07 | 轮询间隔 | 执行详情页运行中状态轮询间隔为 5 秒 |
| NF-08 | API 超时 | Chaos Mesh API 请求超时 10 秒，Coroot API 请求超时 10 秒 |

### 4.4 部署与运维

| 编号 | 需求 | 说明 |
|------|------|------|
| NF-09 | 容器化部署 | 通过 Docker 镜像部署，推送到 Harbor 私有仓库 |
| NF-10 | K8s 部署 | 部署到 ischaos 命名空间，通过 NodePort（30580）对外提供服务 |
| NF-11 | RBAC 权限 | 平台 ServiceAccount 具有 K8s 集群的 Pod/Node/Service/Deployment 只读权限 |
| NF-12 | 环境配置 | 所有外部服务地址通过环境变量配置，支持灵活切换环境 |

### 4.5 安全性

| 编号 | 需求 | 说明 |
|------|------|------|
| NF-13 | K8s 最小权限 | 仅申请必要的只读权限（get/list/watch），不包含写权限 |
| NF-14 | 参数验证 | 所有 API 接口对请求参数进行校验，拒绝非法输入 |
| NF-15 | 运行中保护 | 运行中的执行不允许直接删除，需先停止 |

## 5. 系统边界与约束

### 5.1 系统边界

```
                     ISChaos-sub 平台
┌──────────────────────────────────────────────────┐
│                                                  │
│  浏览器（Vue 3 + AntV X6 + ECharts）              │
│       ↕ HTTP API                                 │
│  Flask 后端（Jinja2 + SQLite）                    │
│       ↕                 ↕                ↕       │
│  Chaos Mesh API    Coroot API    Kubernetes API   │
│                                                  │
└──────────────────────────────────────────────────┘
          ↓                ↓              ↓
   Chaos Mesh       Coroot Server    K8s API Server
   (故障注入)        (可观测数据)      (集群拓扑)
          ↓
   Train-Ticket 微服务集群
   (train-ticket namespace)
```

### 5.2 外部依赖

| 依赖 | 用途 | 通信方式 |
|------|------|---------|
| Chaos Mesh Dashboard | 故障注入和 Workflow 管理 | REST API (HTTP) |
| Coroot | Traces、Metrics、Logs 观测数据 | REST API (HTTP, Session 认证) |
| Kubernetes API Server | Pod/Service/Node 部署拓扑 | Kubernetes Python Client (HTTPS) |
| Train-Ticket | 故障注入目标系统 | 通过 Chaos Mesh 间接操作 |

### 5.3 约束条件

1. **目标系统固定**：当前仅支持 Train-Ticket 微服务系统（train-ticket 命名空间）
2. **单实例部署**：使用 SQLite 数据库，不支持多实例水平扩展
3. **前后端不分离**：所有页面通过 Jinja2 模板渲染，前端框架通过 CDN 引入
4. **无用户认证**：平台本身不包含用户登录/权限管理功能
5. **故障类型固定**：支持 10 种预定义故障类型，不支持自定义故障

## 6. 验收标准

### 6.1 功能验收

| 编号 | 验收项 | 验收标准 |
|------|--------|---------|
| AC-01 | 接口列表展示 | 打开 /interfaces 页面，能看到 15 个业务接口的完整信息 |
| AC-02 | 拓扑图渲染 | 点击接口进入拓扑页，能看到分层布局的调用链图，服务/数据库节点正确区分 |
| AC-03 | 故障配置 | 点击服务节点，弹出配置弹窗，选择故障类型后动态展示对应参数表单 |
| AC-04 | 计划管理 | 能创建、查看、编辑、删除演练计划，预览 Workflow JSON |
| AC-05 | 执行演练 | 点击执行后，Chaos Mesh 中成功创建 Workflow，平台显示 running 状态 |
| AC-06 | 状态同步 | 执行过程中，状态时间线实时更新；Workflow 完成后自动标记 completed |
| AC-07 | 停止演练 | 点击停止按钮，Chaos Mesh Workflow 被终止，状态更新为 completed |
| AC-08 | 删除记录 | 能删除已完成/失败的执行记录，运行中的记录删除被拒绝 |
| AC-09 | 故障互斥 | 同一服务连续执行两次，第二次返回 409 冲突 |
| AC-10 | 指标图表 | 执行详情页展示与故障类型对应的时序指标图表 |
| AC-11 | 数据下载 | Traces/Metrics/Logs 三个 Tab 各自独立下载对应数据的 JSON 文件 |
| AC-12 | K8s 拓扑 | /api/v1/topology 返回真实的 Pod/Service/Node 部署关系 |

### 6.2 非功能验收

| 编号 | 验收项 | 验收标准 |
|------|--------|---------|
| AC-13 | 超时保护 | 故障注入超过 3 倍 duration 后，执行自动标记为 failed |
| AC-14 | 锁释放 | 演练完成/超时/停止后，服务锁被正确释放，可以再次执行同一服务的故障 |
| AC-15 | 容器部署 | Docker 镜像构建成功，K8s 部署后平台通过 NodePort 30580 可访问 |
| AC-16 | RBAC 权限 | 集群内部署模式下，平台能正常查询 Pod/Node/Service 信息 |
