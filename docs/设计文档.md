### 1.2 技术选型

| 组件 | 技术 | 说明 |
|------|------|------|
| Web 框架 | Flask 3.1.0 | 轻量，前后端一体（Jinja2 模板） |
| 数据库 | SQLite | 单文件嵌入式，适合单实例部署 |
| 前端框架 | Vue 3 (CDN) | 响应式 UI，无需构建流程 |
| 拓扑可视化 | AntV X6 v2 + Dagre | 有向图分层布局 |
| 图表 | ECharts | 时序数据可视化 |
| CSS | TailwindCSS (CDN) | 原子化 CSS，快速开发 |
| 故障注入 | Chaos Mesh | 通过 Dashboard REST API 管理 Workflow |
| 可观测性 | Coroot | Traces/Metrics/Logs 数据采集 |
| 集群感知 | kubernetes Python client | Pod/Node/Service 部署拓扑 |
| 部署 | Docker + Kubernetes | 容器化部署到 K8s 集群 |
| WSGI 服务器 | Gunicorn | 生产环境 2 workers |

### 2.2 表结构详细说明

#### interface_topology — 接口拓扑表

存储 15 个核心业务接口及其调用链拓扑数据。

| 列名 | 类型 | 约束 | 说明 |
|------|------|------|------|
| id | INTEGER | PK, AUTO | 自增主键 |
| name | TEXT | NOT NULL | 接口名称（如 "Login"） |
| url | TEXT | NOT NULL | 请求 URL 路径 |
| span_name | TEXT | NOT NULL | Coroot SpanName（如 "POST /api/v1/...") |
| method | TEXT | NOT NULL, DEFAULT 'POST' | HTTP 方法 |
| total_requests | INTEGER | NOT NULL, DEFAULT 0 | 压测总请求数 |
| topology_json | TEXT | NOT NULL | 调用链拓扑 JSON（nodes + edges） |
| created_at | TEXT | NOT NULL, DEFAULT datetime('now') | 创建时间 |

**topology_json 结构：**

```json
{
  "nodes": [
    {"id": "ts-travel-service", "type": "service", "label": "ts-travel-service", "root": true},
    {"id": "ts-route-service", "type": "service", "label": "ts-route-service", "root": false},
    {"id": "ts-travel-mongo", "type": "db", "label": "ts-travel-mongo", "root": false}
  ],
  "edges": [
    {"source": "ts-travel-service", "target": "ts-route-service", "label": "HTTP"},
    {"source": "ts-travel-service", "target": "ts-travel-mongo", "label": "MONGODB"}
  ]
}
```

#### drill_plan — 演练计划表

| 列名 | 类型 | 约束 | 说明 |
|------|------|------|------|
| id | INTEGER | PK, AUTO | 自增主键 |
| name | TEXT | NOT NULL | 计划名称 |
| interface_id | INTEGER | NOT NULL, FK | 关联 interface_topology.id |
| fault_type | TEXT | NOT NULL | 故障类型（10 种之一） |
| target_service | TEXT | NOT NULL | 注入目标服务名 |
| fault_params | TEXT | NOT NULL, DEFAULT '{}' | 故障参数 JSON |
| duration | TEXT | NOT NULL, DEFAULT '30s' | 故障持续时间 |
| status | TEXT | NOT NULL, DEFAULT 'draft' | draft / ready / archived |
| created_at | TEXT | NOT NULL, DEFAULT datetime('now') | 创建时间 |
| updated_at | TEXT | NOT NULL, DEFAULT datetime('now') | 更新时间 |

#### drill_execution — 演练执行记录表

| 列名 | 类型 | 约束 | 说明 |
|------|------|------|------|
| id | INTEGER | PK, AUTO | 自增主键 |
| plan_id | INTEGER | NOT NULL, FK | 关联 drill_plan.id |
| workflow_name | TEXT | | Chaos Mesh Workflow 名称 |
| status | TEXT | NOT NULL, DEFAULT 'pending' | pending / running / collecting / completed / failed |
| started_at | TEXT | | 执行开始时间（ISO 8601） |
| finished_at | TEXT | | 执行结束时间 |
| fault_inject_at | TEXT | | 故障实际注入时间 |
| fault_end_at | TEXT | | 故障实际结束时间 |
| result_json | TEXT | NOT NULL, DEFAULT '{}' | 观测数据摘要 JSON |
| error_message | TEXT | | 错误信息 |
| created_at | TEXT | NOT NULL, DEFAULT datetime('now') | 记录创建时间 |

#### service_fault_lock — 服务故障互斥锁表

| 列名 | 类型 | 约束 | 说明 |
|------|------|------|------|
| id | INTEGER | PK, AUTO | 自增主键 |
| service_name | TEXT | NOT NULL, UNIQUE | 被锁定的服务名 |
| execution_id | INTEGER | NOT NULL, FK | 持有锁的执行记录 ID |
| locked_at | TEXT | NOT NULL, DEFAULT datetime('now') | 锁获取时间 |

**互斥机制：** 利用 SQLite 的 UNIQUE 约束实现原子性互斥。`INSERT` 同一 `service_name` 时触发 `IntegrityError`，即加锁失败。

## 3. API 接口详细定义

### 3.1 统一响应格式

所有 API 均返回以下 JSON 格式：

```json
{
  "code": 200,
  "message": "success",
  "data": { ... }
}
```

错误响应：

```json
{
  "code": 404,
  "message": "计划不存在",
  "data": null
}
```

### 3.2 页面渲染路由

| 方法 | 路径 | 模板 | 说明 |
|------|------|------|------|
| GET | `/` | base.html | 首页 |
| GET | `/interfaces` | interfaces.html | 接口列表页 |
| GET | `/topology/<interface_id>` | topology.html | 拓扑详情页 |
| GET | `/plans` | plans.html | 计划列表页 |
| GET | `/executions` | executions.html | 执行列表页 |
| GET | `/executions/<execution_id>` | execution.html | 执行详情页 |

### 3.3 接口拓扑 API

#### GET /api/interfaces — 获取接口列表

**响应示例：**

```json
{
  "code": 200,
  "message": "success",
  "data": [
    {
      "id": 1,
      "name": "Login",
      "url": "/api/v1/users/login",
      "span_name": "POST /api/v1/users/login",
      "method": "POST",
      "total_requests": 5000,
      "service_count": 3,
      "edge_count": 4,
      "created_at": "2026-03-10 10:00:00"
    }
  ]
}
```

> 注：列表接口不返回 topology_json，减少响应体积。

#### GET /api/interfaces/\<interface_id\> — 获取接口详情（含拓扑）

**响应示例：**

```json
{
  "code": 200,
  "message": "success",
  "data": {
    "id": 1,
    "name": "Login",
    "url": "/api/v1/users/login",
    "span_name": "POST /api/v1/users/login",
    "method": "POST",
    "total_requests": 5000,
    "topology_json": {
      "nodes": [
        {"id": "ts-auth-service", "type": "service", "label": "ts-auth-service", "root": true},
        {"id": "ts-auth-mongo", "type": "db", "label": "ts-auth-mongo", "root": false}
      ],
      "edges": [
        {"source": "ts-auth-service", "target": "ts-auth-mongo", "label": "MONGODB"}
      ]
    },
    "created_at": "2026-03-10 10:00:00"
  }
}
```

### 3.4 演练计划 API

#### GET /api/plans — 获取所有计划

**响应示例：**

```json
{
  "code": 200,
  "message": "success",
  "data": [
    {
      "id": 1,
      "name": "ts-travel-service 网络延迟",
      "interface_id": 3,
      "fault_type": "network_delay",
      "target_service": "ts-travel-service",
      "fault_params": {"latency": "200ms", "jitter": "50ms", "correlation": "0"},
      "duration": "30s",
      "status": "ready",
      "created_at": "2026-03-11 09:00:00",
      "updated_at": "2026-03-11 09:00:00"
    }
  ]
}
```

#### GET /api/plans/\<plan_id\> — 获取单个计划

响应格式同上，返回单条记录。

#### POST /api/plans — 创建计划

**请求体：**

```json
{
  "name": "ts-travel-service 网络延迟",
  "interface_id": 3,
  "fault_type": "network_delay",
  "target_service": "ts-travel-service",
  "fault_params": {"latency": "200ms", "jitter": "50ms"},
  "duration": "30s",
  "status": "ready"
}
```

**必填字段：** name, interface_id, fault_type, target_service

**响应：** 返回创建后的完整计划对象。

#### PUT /api/plans/\<plan_id\> — 更新计划

**请求体（部分更新）：**

```json
{
  "status": "ready",
  "duration": "60s"
}
```

#### DELETE /api/plans/\<plan_id\> — 删除计划

**响应：** `{"code": 200, "message": "success", "data": null}`

#### GET /api/plans/\<plan_id\>/workflow — 预览 Workflow

**响应示例：**

```json
{
  "code": 200,
  "message": "success",
  "data": {
    "workflow_name": "ischaos-1-1710000000",
    "workflow_json": {
      "apiVersion": "chaos-mesh.org/v1alpha1",
      "kind": "Workflow",
      "metadata": {"name": "ischaos-1-1710000000", "namespace": "chaos-mesh"},
      "spec": {
        "entry": "fault-1-1710000000",
        "templates": [...]
      }
    }
  }
}
```

#### POST /api/plans/\<plan_id\>/execute — 执行计划

**前置条件：** 计划 status 为 ready

**响应（成功）：**

```json
{
  "code": 200,
  "message": "success",
  "data": {
    "id": 5,
    "plan_id": 1,
    "workflow_name": "ischaos-1-1710000000",
    "status": "running",
    "started_at": "2026-03-12T07:00:00Z",
    "fault_inject_at": "2026-03-12T07:00:00Z",
    "finished_at": null,
    "fault_end_at": null,
    "result_json": {},
    "error_message": null,
    "created_at": "2026-03-12 07:00:00"
  }
}
```

**响应（服务锁冲突）：**

```json
{
  "code": 409,
  "message": "服务 'ts-travel-service' 已有活跃故障",
  "data": null
}
```

### 3.5 执行管理 API

#### GET /api/executions — 获取所有执行记录

**响应示例：**

```json
{
  "code": 200,
  "message": "success",
  "data": [
    {
      "id": 5,
      "plan_id": 1,
      "plan_name": "ts-travel-service 网络延迟",
      "fault_type": "network_delay",
      "target_service": "ts-travel-service",
      "workflow_name": "ischaos-1-1710000000",
      "status": "completed",
      "started_at": "2026-03-12T07:00:00Z",
      "finished_at": "2026-03-12T07:01:05Z",
      "fault_inject_at": "2026-03-12T07:00:00Z",
      "fault_end_at": "2026-03-12T07:00:35Z",
```

> 注：列表接口额外附带 plan_name、fault_type、target_service 字段。

#### GET /api/executions/\<execution_id\> — 获取单条执行

响应同列表中的单条记录（不含 plan_name 等附加字段）。

#### GET /api/executions/\<execution_id\>/status — 查询实时状态

自动同步 Chaos Mesh Workflow 状态并更新本地记录。

**响应示例：**

```json
{
  "code": 200,
  "message": "success",
  "data": {
    "execution": { "id": 5, "status": "running", ... },
    "chaos_status": {
      "nodeInfoList": [
        {
          "stepSpanList": [
            {"startTime": "2026-03-12T07:00:00Z", "endTime": "2026-03-12T07:00:01Z"},
            {"startTime": "2026-03-12T07:00:01Z", "endTime": "1970-01-01T00:00:00Z"},
            {"startTime": "1970-01-01T00:00:00Z", "endTime": "1970-01-01T00:00:00Z"}
          ]
        }
      ]
    }
  }
}
```

#### PUT /api/executions/\<execution_id\>/stop — 停止执行

**前置条件：** 执行 status 为 running

**响应：** 返回更新后的执行记录（status 变为 completed）。

#### DELETE /api/executions/\<execution_id\> — 删除执行记录

**前置条件：** 执行 status 不为 running

**响应：** `{"code": 200, "message": "success", "data": null}`

**错误响应（运行中）：** `{"code": 400, "message": "运行中的执行不能删除，请先停止"}`

#### GET /api/executions/\<execution_id\>/metrics — 获取故障指标

**查询参数（均可选）：**
- `from`: 起始时间戳（毫秒）
- `to`: 结束时间戳（毫秒）

**默认时间范围：** fault_inject_at 前 10 分钟 ~ fault_end_at 后 10 分钟

**响应示例：**

```json
{
  "code": 200,
  "message": "success",
  "data": {
    "fault_type": "network_delay",
    "metric_name": "Network RTT",
    "metric_unit": "ms",
    "description": "服务间网络往返时间",
    "time_range": {"from": 1710000000000, "to": 1710001200000},
    "threshold": null,
    "series": [
      {
        "name": "ts-travel-service → ts-route-service",
        "data": [[1710000000000, 1.5], [1710000060000, 205.3], ...]
      }
    ]
  }
}
```

## 4. 核心流程时序图

### 4.1 执行演练流程

```
用户          plans.html        plan_routes         execution_model    fault_lock      workflow_builder    chaos_client       Chaos Mesh
 │               │                   │                    │               │                  │                 │                │
 │──[执行]──────►│                   │                    │               │                  │                 │                │
 │               │──POST /execute───►│                    │               │                  │                 │                │
```

## 5. 故障类型与 Workflow 模板映射

### 5.1 Workflow 结构（FaultConfig 格式）

所有故障类型生成统一的 Workflow 结构：

```json
{
  "apiVersion": "chaos-mesh.org/v1alpha1",
  "kind": "Workflow",
  "metadata": {
    "name": "ischaos-{plan_id}-{timestamp}",
    "namespace": "chaos-mesh"
  },
  "spec": {
    "entry": "fault-{plan_id}-{timestamp}",
    "templates": [
      {
        "name": "fault-{plan_id}-{timestamp}",
        "templateType": "FaultConfig",
        "deadline": "{duration × 2}",
        "list": [
          {"type": "Inject", "chaosType": "...", "{chaosType}": { ... }},
          {"type": "Wait", "deadline": "{duration}"},
          {"type": "Recover", "index": 0}
        ]
      }
    ]
  }
}
```

### 5.2 故障类型详细映射

#### NetworkChaos (网络故障)

| 故障类型 | action | Inject spec 关键字段 |
|----------|--------|---------------------|
| network_delay | delay | `delay: {latency, jitter, correlation}`, `direction`, `selector` |
| network_loss | loss | `loss: {loss, correlation}`, `direction`, `selector` |
| network_partition | partition | `direction: "both"`, `selector` |

**selector 结构（通用）：**
```json
{
  "namespaces": ["train-ticket"],
  "labelSelectors": {"app": "{target_service}"}
}
```

#### PodChaos (Pod 故障)

| 故障类型 | action | Inject spec 关键字段 |
|----------|--------|---------------------|
| pod_failure | pod-failure | `mode: "all"`, `selector` |
| pod_kill | pod-kill | `mode: "all"`, `gracePeriod`, `selector` |

#### StressChaos (压力故障)

| 故障类型 | stressors 字段 |
|----------|---------------|
| stress_cpu | `cpu: {workers, load}` |
| stress_mem | `memory: {workers, size}` |

#### DNSChaos

| 故障类型 | action | 关键字段 |
|----------|--------|---------|
| dns_error | error | `patterns: [...]`, `selector` |

#### PhysicalMachineChaos (物理机故障)

| 故障类型 | action | 关键字段 |
|----------|--------|---------|
| node_cpu | stress-cpu | `address: [...]`, `stress-cpu: {workers, load}` |
| node_mem | stress-mem | `address: [...]`, `stress-mem: {size}` |

> 注：PhysicalMachineChaos 不使用 selector，直接指定物理机地址。

### 5.3 故障指标关联

| 故障类型 | 观测指标 | 数据来源 |
|----------|---------|---------|
| network_loss | TCP Retransmissions | Coroot App |
| network_delay | Network RTT | Coroot App |
| network_partition | Active TCP Connections | Coroot App |
| pod_failure | Instance Count | Coroot App |
| pod_kill | Container Restarts | Coroot App |
| stress_cpu | Container CPU Usage | Coroot App |
| stress_mem | Container Memory RSS | Coroot App |
| dns_error | DNS Errors | Coroot DNS App (CoreDNS) |
| node_cpu | Node CPU Usage | Coroot Node |
| node_mem | Node Memory Usage | Coroot Node |

### 6.1 整体布局

```
┌──────────────────────────────────────────────┐
│ ┌──────────┐ ┌─────────────────────────────┐ │
│ │          │ │  Header (页面标题)           │ │
│ │  侧边栏   │ │                             │ │
│ │          │ ├─────────────────────────────┤ │
│ │ 链路接口  │ │                             │ │

│ │          │ │                             │ │

│ │ v0.1.0   │ │                             │ │
│ └──────────┘ └─────────────────────────────┘ │
└──────────────────────────────────────────────┘
```

### 6.2 页面功能

| 页面 | 文件 | 核心功能 |
|------|------|---------|
| 接口列表 | interfaces.html | 表格展示 15 个接口，点击行跳转拓扑页 |
| 拓扑详情 | topology.html | AntV X6 图渲染调用链，点击节点配置故障 |
| 计划列表 | plans.html | 表格展示计划，支持预览 Workflow、执行、删除 |
| 执行列表 | executions.html | 表格展示执行记录，支持查看详情、删除 |
| 执行详情 | execution.html | 状态时间线、计划信息、指标图表、数据收集 |

### 6.3 拓扑渲染 (topology.js)

- 使用 AntV X6 v2 自定义 HTML 节点
- 3 种节点类型：service-node（白色）、root-node（蓝色边框）、db-node（虚线边框）
- Dagre 分层布局（rankdir=TB, nodesep=100, ranksep=70）
- Manhattan 路由 + 圆角拐角
- 交互：悬停高亮上下游路径（BFS），点击节点触发回调
- 缩放范围：0.3x ~ 3x

### 6.4 指标图表 (ECharts)

- 时间轴折线图，支持多系列叠加
- X 轴：时间（自动格式化 HH:mm:ss）
- Y 轴：指标值 + 单位（自动适配）
- tooltip 浮窗显示时间和数值
- 观测窗口：故障注入前 10 分钟 ~ 故障恢复后 10 分钟
