> 版本：v1.0 | 更新日期：2026-03-12

## 1. 系统访问方式

### 1.1 本地开发环境

```bash
# 安装依赖
cd ischaos-platform
pip install -r requirements.txt

# 初始化数据库（首次运行）
python init_db.py ../docs/topology_report.md

# 启动开发服务器
export KUBECONFIG=~/.kube/coroot-config
python app.py
```

访问地址：**http://localhost:5001**

### 1.2 K8s 集群环境

平台部署在 `ischaos` 命名空间，通过 NodePort 暴露。

访问地址：**http://\<任意节点IP\>:30580**

### 1.3 数据持久化

平台使用 NFS PVC 持久化 SQLite 数据库，Pod 重启/重建不会丢失数据。

| 项目 | 值 |
|------|-----|
| NFS 服务器 | 192.168.0.197 |
| NFS 路径 | /data/share/ischaos-ischaos-data-pvc-c94d5c2c-fdbe-4401-9bd8-384b8def4f83 |
| Pod 内路径 | /app/data/ischaos.db |

**首次部署需初始化数据库：**

```bash
POD=$(kubectl get pods -n ischaos -l app=ischaos-platform -o jsonpath='{.items[0].metadata.name}')
kubectl cp topology_report.md ischaos/$POD:/tmp/topology_report.md
kubectl exec $POD -n ischaos -- python init_db.py /tmp/topology_report.md
```

## 2. 功能操作指南

### 2.1 查看链路接口列表

**路径：** 侧边栏 → **链路接口**（或直接访问 `/interfaces`）

页面展示 15 个核心业务接口，每个接口显示：

| 列 | 说明 |
|----|------|
| # | 接口编号 |
| 接口名称 | 如 "Login"、"QueryTravel" |
| HTTP 方法 | GET / POST / PUT / DELETE（彩色标签） |
| URL | 接口请求路径 |
| 服务数 | 调用链中涉及的微服务数量 |
| 边数 | 调用链中的调用关系数量 |
| 请求数 | 压测期间的总请求数 |

**操作：** 点击任意行，进入该接口的调用链拓扑页。

### 2.2 查看链路拓扑

**路径：** 接口列表 → 点击行（或直接访问 `/topology/<接口ID>`）

页面顶部显示接口基本信息（名称、URL、SpanName），下方为拓扑可视化区域。

#### 拓扑图说明

- **白色方框**：普通服务节点（微服务）
- **蓝色边框方框**：根节点（入口服务）
- **虚线方框**：数据库节点（MongoDB）
- **实线箭头**：HTTP 调用关系
- **虚线箭头**：数据库访问关系

#### 交互操作

| 操作 | 效果 |
|------|------|
| 鼠标滚轮 | 缩放拓扑图 |
| 拖拽空白区域 | 平移画布 |
| 悬停在节点上 | 高亮该节点的上下游调用路径，其他节点变暗 |
| 点击服务节点 | 弹出故障配置弹窗 |
| 适应画面按钮 | 自动缩放至全部节点可见 |
| 放大/缩小按钮 | 手动调整缩放级别 |

### 2.3 配置故障演练

**路径：** 拓扑页 → 点击服务节点 → 弹出故障配置弹窗

#### 配置步骤

1. **点击目标服务节点**：在拓扑图中点击要注入故障的服务（白色或蓝色方框），弹出配置弹窗。

2. **填写计划名称**：输入描述性名称，如 "ts-travel-service 网络延迟测试"。

3. **选择故障类型**：从下拉菜单选择 10 种故障类型之一（详见第 3 节）。选择后会动态显示对应的参数表单。

4. **配置故障参数**：根据故障类型填写参数。每个参数有默认值，可直接使用或修改。

5. **设置持续时间**：输入故障持续时间，格式为数字+单位，如 `30s`（30 秒）、`5m`（5 分钟）。

6. **点击"创建计划"**：提交后创建 `ready` 状态的演练计划，弹窗显示成功提示后自动关闭。

### 2.4 管理演练计划

**路径：** 侧边栏 → **演练计划**（或直接访问 `/plans`）

页面展示所有已创建的演练计划。

#### 计划状态

| 状态 | 说明 | 可执行 |
|------|------|--------|
| draft | 草稿，初始状态 | 否 |
| ready | 就绪，可以执行 | 是 |
| archived | 归档，不再使用 | 否 |

> 从拓扑页创建的计划默认为 `ready` 状态，可直接执行。

#### 操作按钮

| 按钮 | 说明 |
|------|------|
| **Workflow** | 预览将提交给 Chaos Mesh 的 Workflow JSON |
| **执行** | 触发演练执行（仅 ready 状态可用） |
| **删除** | 删除计划（弹出确认对话框） |

#### 执行演练

1. 在计划列表中，找到状态为 `ready` 的计划。
2. 点击 **"执行"** 按钮。
3. 系统自动完成以下操作：
   - 构建 Chaos Mesh Workflow JSON
   - 创建执行记录
   - 检查服务故障互斥锁（同一服务只允许一个活跃故障）
   - 向 Chaos Mesh 提交 Workflow
4. 执行成功后，页面显示成功提示和执行记录链接。
5. 点击链接跳转到执行详情页。

**可能的错误：**
- "计划状态为 'draft'，只有 'ready' 状态的计划才能执行" → 需先将计划状态改为 ready
- "服务 'xxx' 已有活跃故障" → 同一服务已有运行中的故障，需等待完成

### 2.5 查看执行记录列表

**路径：** 侧边栏 → **演练记录**（或直接访问 `/executions`）

展示所有执行记录，包含：

| 列 | 说明 |
|----|------|
| # | 执行编号 |
| 计划名称 | 关联的演练计划名称 |
| 故障类型 | 彩色标签显示故障类型 |
| 目标服务 | 故障注入的目标服务 |
| 状态 | 当前执行状态（见下表） |
| 开始时间 | 演练开始时间 |
| 结束时间 | 演练结束时间 |
| 操作 | 详情 / 删除按钮 |

#### 执行状态说明

| 状态 | 标签颜色 | 说明 |
|------|---------|------|
| pending | 灰色 | 等待中，已创建但未开始 |
| running | 蓝色 | 运行中，故障正在注入 |
| collecting | 黄色 | 数据收集中 |
| completed | 绿色 | 已完成 |
| failed | 红色 | 失败（超时或异常） |

#### 删除执行记录

- 点击 **"删除"** 按钮（仅非 running 状态显示）
- 弹出确认对话框，确认后永久删除
- 运行中的执行不允许直接删除，需先停止

### 2.6 查看执行详情与状态

**路径：** 执行列表 → 点击 **"详情"**（或直接访问 `/executions/<id>`）

执行详情页包含以下区域：

#### 状态时间线

页面顶部显示 5 个状态节点的进度：

```
[已创建] ──► [运行中] ──► [数据收集] ──► [已完成]
```

当前状态高亮显示，已过状态打勾，失败状态显示红色。

#### 计划信息面板

显示关联计划的详细信息：名称、故障类型、目标服务、持续时间、关联接口。

#### Chaos Mesh 工作流状态

显示 Workflow 的 3 个步骤执行状态：

| 步骤 | 说明 |
|------|------|
| Inject | 故障注入开始/结束时间 |
| Wait | 等待期（故障持续）开始/结束时间 |
| Recover | 故障恢复开始/结束时间 |

#### 操作按钮

| 按钮 | 条件 | 说明 |
|------|------|------|
| **停止执行** | 仅 running 状态 | 调用 Chaos Mesh 停止 Workflow，释放故障锁 |
| **刷新** | 任何时候 | 手动刷新状态（running 时自动每 5 秒刷新） |

#### 故障指标图表

显示与故障类型对应的可观测指标时序图（ECharts 折线图）。

- **观测窗口**：故障注入前 10 分钟 ~ 故障恢复后 10 分钟
- **自动获取**：进入详情页后自动加载指标数据
- **手动刷新**：点击刷新按钮重新获取

指标对应关系参见第 3 节。

### 2.7 收集观测数据

执行详情页底部提供三个数据收集标签页：

| Tab | 数据类型 | 说明 |
|-----|---------|------|
| **Traces** | 分布式调用链 | 目标服务在故障期间的请求调用链 |
| **Metrics** | 监控指标 | CPU、内存、网络、延迟等时序指标 |
| **Logs** | 结构化日志 | 目标服务在故障期间的应用日志 |

#### 操作步骤

1. 切换到目标 Tab（Traces / Metrics / Logs）。
2. 点击 **"下载 JSON"** 按钮。
3. 系统自动：
   - 以故障注入前后各 10 分钟为时间窗口
   - 调用对应的观测数据 API
   - 将数据打包为 JSON 文件并下载
4. 文件命名格式：`execution_{id}_{tab类型}.json`

每个 Tab 独立管理数据，切换 Tab 不会影响已获取的数据。

## 3. 故障类型说明

### 3.1 网络故障

#### network_delay — 网络延迟

**用途：** 模拟网络延迟，验证服务在高延迟条件下的超时处理和降级能力。

| 参数 | 说明 | 默认值 | 示例 |
|------|------|--------|------|
| latency | 延迟时长 | 200ms | 500ms |
| jitter | 抖动幅度 | 0ms | 50ms |
| correlation | 延迟相关性百分比 | 0 | 50 |

**观测指标：** Network RTT（网络往返时间）

---

#### network_loss — 网络丢包

**用途：** 模拟网络丢包，验证服务的重试机制和容错能力。

| 参数 | 说明 | 默认值 | 示例 |
|------|------|--------|------|
| loss | 丢包率百分比 | 50 | 80 |
| correlation | 丢包相关性百分比 | 0 | 25 |

**观测指标：** TCP Retransmissions（TCP 重传次数）

---

#### network_partition — 网络隔离

**用途：** 模拟网络分区，验证服务在完全断网条件下的行为和恢复能力。

| 参数 | 说明 | 默认值 | 可选值 |
|------|------|--------|-------|
| direction | 隔离方向 | both | both（双向）/ to（出方向）/ from（入方向） |

**观测指标：** Active TCP Connections（活跃 TCP 连接数）

---

### 3.2 Pod 故障

#### pod_failure — Pod 故障

**用途：** 使目标 Pod 不可用（不终止进程），验证服务的健康检查和流量转移能力。

**参数：** 无额外参数。

**观测指标：** Instance Count（实例数量变化）

---

#### pod_kill — Pod 终止

**用途：** 终止目标 Pod 进程，验证 Kubernetes 的自动重启和服务恢复能力。

| 参数 | 说明 | 默认值 | 示例 |
|------|------|--------|------|
| gracePeriod | 优雅终止等待时间（秒） | 0 | 30 |

**观测指标：** Container Restarts（容器重启次数）

---

### 3.3 压力故障

#### stress_cpu — CPU 压力

**用途：** 对目标 Pod 注入 CPU 压力，验证服务在高 CPU 负载下的性能表现和限流能力。

| 参数 | 说明 | 默认值 | 示例 |
|------|------|--------|------|
| workers | 并发压力工作进程数 | 1 | 2 |
| load | CPU 负载百分比 | 80 | 90 |

**观测指标：** Container CPU Usage（容器 CPU 使用率）

---

#### stress_mem — 内存压力

**用途：** 对目标 Pod 注入内存压力，验证服务的 OOM 处理和内存限制策略。

| 参数 | 说明 | 默认值 | 示例 |
|------|------|--------|------|
| workers | 并发压力工作进程数 | 1 | 2 |
| size | 占用内存大小 | 256MB | 512MB |

**观测指标：** Container Memory RSS（容器内存 RSS 占用）

---

### 3.4 其他故障

#### dns_error — DNS 错误

**用途：** 注入 DNS 解析错误，验证服务在 DNS 异常时的容错和降级行为。

| 参数 | 说明 | 默认值 | 示例 |
|------|------|--------|------|
| patterns | DNS 匹配模式（逗号分隔） | （空） | *.svc.cluster.local |

**观测指标：** DNS Errors（DNS 解析错误数，来源为 CoreDNS）

---

#### node_cpu — 节点 CPU 压力

**用途：** 对物理节点注入 CPU 压力，验证节点级别资源竞争对服务的影响。

| 参数 | 说明 | 默认值 | 示例 |
|------|------|--------|------|
| workers | 并发压力工作进程数 | 1 | 4 |
| load | CPU 负载百分比 | 80 | 95 |
| address | 物理机 Chaosd 地址 | （必填） | http://10.0.0.1:31767 |

**观测指标：** Node CPU Usage（节点 CPU 使用率）

---

#### node_mem — 节点内存压力

**用途：** 对物理节点注入内存压力，验证节点级别内存不足对服务的影响。

| 参数 | 说明 | 默认值 | 示例 |
|------|------|--------|------|
| size | 占用内存大小 | 256MB | 1GB |
| address | 物理机 Chaosd 地址 | （必填） | http://10.0.0.1:31767 |

**观测指标：** Node Memory Usage（节点内存使用率）

## 4. 常见问题 FAQ

### Q1: 执行演练时提示"服务已有活跃故障"怎么办？

**原因：** 同一服务在同一时间只允许存在一个活跃故障（互斥约束）。

**解决：**
1. 等待当前演练自动完成
2. 在执行详情页手动停止当前演练
3. 如果是残留锁（执行已终态但锁未释放），删除对应的执行记录会自动清理锁

---

### Q2: 执行状态一直显示 running 不更新？

**可能原因：**
1. Chaos Mesh Workflow 未正常启动
2. 网络连接问题导致状态同步失败

**解决：**
1. 点击"刷新"按钮手动触发状态同步
2. 检查 Chaos Mesh Dashboard 中 Workflow 的实际状态
3. 如果 Workflow 已异常，手动停止执行

**自动保护：** 执行时长超过计划 duration 的 3 倍后，系统会自动将其标记为超时失败并释放锁。

---

### Q3: 拓扑页面空白，没有图渲染？

**可能原因：**
1. 接口拓扑数据未导入
2. 浏览器不支持或 CDN 资源未加载

**解决：**
1. 确认已运行 `python init_db.py` 导入拓扑数据
2. 检查浏览器控制台是否有 JavaScript 错误
3. 确认网络能访问 CDN（unpkg.com 等）

---

### Q4: 故障指标图表没有数据？

**可能原因：**
1. 执行尚未有 fault_inject_at 时间（故障还未实际注入）
2. Coroot 中没有对应服务的指标数据
3. 时间范围不在 Coroot 数据保留期内

**解决：**
1. 等待执行进入 running 状态且有 fault_inject_at 时间后再查看
2. 在 Coroot 界面确认目标服务有指标数据
3. 手动通过 `from` 和 `to` 参数指定时间范围

---

### Q5: 物理机故障（node_cpu/node_mem）执行失败？

**可能原因：** 目标节点未安装或未启动 Chaosd Agent。

**解决：**
1. 确认目标节点上的 Chaosd Agent 已安装并运行
2. 确认 `address` 参数格式正确（如 `http://10.0.0.1:31767`）
3. 确认 Chaos Mesh 到目标节点的网络连通

---

### Q6: 如何查看 Chaos Mesh 中 Workflow 的原始状态？

**方法：**
1. 在执行详情页的 Chaos Mesh 工作流状态区域可以看到步骤时间信息
2. 也可直接访问 Chaos Mesh Dashboard（默认 http://116.63.51.45:30854）查看 Workflow 详情

---

### Q7: 下载的 JSON 数据文件内容为空？

**可能原因：**
1. 指定的时间范围内没有对应的观测数据
2. Coroot API 连接失败

**解决：**
1. 确认演练已执行一段时间，Coroot 有足够数据
2. 检查 Coroot 服务是否正常运行
3. 查看浏览器控制台网络请求是否有报错

---

### Q8: 平台在 K8s 集群中无法查询 Pod/Node 信息？

**可能原因：** RBAC 权限未正确配置。

**解决：**
1. 确认已应用 `k8s/rbac.yaml`：
   ```bash
   kubectl apply -f k8s/rbac.yaml
   ```
2. 确认 Deployment 中设置了 `serviceAccountName: ischaos-sa`
3. 验证权限：
   ```bash
   kubectl auth can-i list pods --as=system:serviceaccount:ischaos:ischaos-sa
   ```
